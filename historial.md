# Historial de Desarrollo - FleetSync AI

- **2026-03-02**: El proyecto ha sido inicializado. Se ha creado el entorno virtual de Python (`venv/`) y la estructura base de carpetas (`backend/`, `frontend/`, `mcp_server/`).
- **2026-03-02**: Se ha desarrollado el servidor MCP simulado en `mcp_server/server.py`. Este servidor expone tres herramientas que simulan APIs de CAMARA de Nokia (`check_number_verification`, `check_sim_swap`, `verify_location`) para su uso posterior por un agente de IA.
- **2026-03-02**: Se ha creado el cliente IA `backend/ai_agent.py`. Este módulo conecta con el MCP Server de forma asíncrona mediante `stdio_client`, ejecuta las herramientas de verificación, y procesa los resultados a través del modelo Gemini (con fallback integrado en caso de no disponer de API key) para retornar una evaluación en formato JSON indicando un status "SECURE" o "ALERT".
- **2026-03-02**: Se ha desarrollado el frontend en `frontend/app.py` utilizando Streamlit. La UI actúa como un Centro de Mando Logístico permitiendo la entrada de parámetros de telemetría, invocando al Agente de IA al pulsar un botón y renderizando el output JSON como banderas visuales (Verde SECURE, Rojo ALERT).
- **2026-03-02**: Se ha creado un script en la raíz `simulador.py`. Este script sirve para hacer una demo técnica ejecutando directamente llamadas al agente de IA con dos casos de uso: Escenario A (Situación normal donde el sistema reporta "SECURE") y Escenario B (Ataque donde el GPS no coincide con la red CAMARA, reportando "ALERT"). Se puede ejecutar la simulación desde consola usando `python simulador.py`.
- **2026-03-02**: Se han configurado medidas de seguridad y variables de entorno. Se crearon `.env` y `.gitignore` en la raíz. El script `backend/ai_agent.py` ha sido actualizado para cargar dichas variables usando `python-dotenv` (para claves como GEMINI_API_KEY o NOKIA_API_KEY) y se han incorporado bloques `try-except` robustos en la conexión e invocación del servidor MCP, garantizando que una caída del backend o de las herramientas no rompa el dashboard, reportando un estatus UNKNOWN de forma controlada.
- **2026-03-02**: Automatización de ejecución lista. Se ha creado el script `start.py` en la raíz del proyecto. Este orquestador ("Botón Rojo") automatiza la secuencia de arranque para los jueces del Hackathon, levantando en segundo plano la infraestructura MCP y ejecutando automáticamente el Dashboard de Streamlit en el puerto por defecto (8501).
- **2026-03-02**: Integración real con la API de Nokia Network as Code (NaC). El servidor `mcp_server/server.py` fue modificado para usar la librería `requests` y enviar peticiones reales al sandbox de Nokia (`/api/location-verification/v0/verify`). Se inyectó la clave Bearer desde la variable de entorno y se estructuró el payload con número, lat, lon y maxAge. Se implementó el manejo de errores estricto para retornar JSON descriptivo al AI si ocurre 401 (Unauthorized) o 500 (Server Error).
- **2026-03-02**: Ingeniería de System Prompt del Agente IA. Se ha inyectado una instrucción de sistema de grado militar en `backend/ai_agent.py` para anclar las decisiones de Gemini a los resultados de las herramientas Telco (especialmente la de red/NaC), asegurando que devuelva un estatus `ALERT` si existe discrepancia y mitigando alucinaciones. También se asegura la respuesta en formato JSON e inyecta los "network_logs" crudos al objeto devuelto para que el frontend los pueda consumir.
- **2026-03-02**: Puliendo la Demo Visual y de Rendimiento para el Jurado. La UI principal en `frontend/app.py` ha sido completamente rediseñada a layout 'wide' con dos columnas (`st.columns`). En la izquierda, los controles del simulador, y en la derecha, el log de la IA. Destaca un nuevo componente "Consola de Red" que expone el JSON real devuelto por la API de Nokia y, a continuación, el banner "estado seguro" o "alerta detectada" mostrando el verdadero poder de la integración visual a un juez.
